# Kickpact — technical documentation (TxLINE hackathon)

## Core idea

Group prediction pools on Solana whose **only settlement authority is a TxLINE Merkle proof**. Friends escrow equal kUSD stakes on a World Cup fixture and pick home/draw/away; after full time **anyone** submits TxLINE's proof of the final goal counts and the `kickpact` program CPIs into TxLINE's `validate_stat_v2` to confirm the claimed outcome against the Merkle roots TxODDS anchors on-chain. Winners split the pot self-serve. No custodian, no admin key, no trusted resolver — the keeper bot is a *courier*, not an *authority*.

## Architecture

```
┌─────────────┐   REST + SSE    ┌──────────────────────────┐
│ TxLINE devnet│ ───────────────▶│ mobile app (Expo RN)     │─┐
│  fixtures    │                 │ dashboard (Next.js)      │ │ txs (create/join/claim)
│  scores      │                 │ keeper (bun daemon)      │ │
│  odds        │                 └──────────────────────────┘ ▼
│  proofs      │                          │ settle(outcome, proof)
└──────┬──────┘                           ▼
       │ insert_scores_root      ┌──────────────────────────┐   CPI validate_stat_v2
       └────────────────────────▶│ Solana devnet            │◀────────────────────────┐
                                 │  txoracle 6pW64g…yP2J    │                         │
                                 │  kickpact 4tAPD5…gWDa ───┴─────────────────────────┘
                                 └──────────────────────────┘
```

**On-chain (Anchor 0.32, [`apps/solana/programs/kickpact/src/lib.rs`](../apps/solana/programs/kickpact/src/lib.rs)):**

- `initialize` — creates the kUSD mint (6 dp SPL, PDA mint authority) + pool registry
- `faucet(amount)` — testnet faucet, ≤1,000 kUSD per call
- `create_pool(fixture_id, stake, deadline_ms, kickoff_ms, pick)` — escrows the creator's stake in the pool PDA's ATA; `kickoff_ms` is the fixture's TxLINE `StartTime` and anchors settlement finality; `deadline_ms` is the join cutoff
- `join_pool(pick)` — equal stake, before the deadline
- `settle(outcome, payload)` — **permissionless**; validates the proof shape (fixture id matches, exactly statKeys 1 & 2, finality: phase `Ended(5)` OR `payload.ts ≥ kickoff + 105min`), pins `daily_scores_merkle_roots` to the oracle PDA derived from the proof's own epoch day (`["daily_scores_roots", epochDay u16 LE]`), **builds the 1X2 predicate on-chain** from the claimed outcome (binary subtract over stat indices), CPIs into `validate_stat_v2`, and requires `true`
- `claim` — winners split `stake × members / winners`; if `winners == 0` every member refunds
- `refund_expired` — 48h after kickoff with no settlement → self-serve refunds

The CPI client is generated by Anchor's `declare_program!` from TxLINE's published IDL ([`apps/solana/idls/txoracle.json`](../apps/solana/idls/txoracle.json)) — no hand-rolled account metas.

**Deterministic tests** ([`apps/solana/tests/kickpact.test.ts`](../apps/solana/tests/kickpact.test.ts)): `anchor test` starts a local validator that **clones the devnet txoracle program and the daily-roots PDA for epoch day 20649**, then runs the whole lifecycle against the committed England 1–2 Argentina proof: a *wrong* outcome is refuted by the oracle (`OracleRefuted`), the true outcome settles, winners split, the vault drains to zero.

## Design decisions worth judging

1. **The strategy is never taken from the caller.** `settle` accepts only an outcome claim; the winning predicate is constructed inside the program. This closes the "prove something true but irrelevant" hole — the oracle call can only ever attest the exact 1X2 question the pool asked.
2. **Finality without trusting the clock of the feed.** TxLINE's last record for a fixture can predate the `Ended` flip (our semifinal capture stops at the 93'). Settlement accepts phase 5 **or** a proof stamped ≥ 105 minutes after the pool's `kickoff_ms` (which the client sets from TxLINE's own `StartTime`).
3. **Epoch-day pinning.** The daily-roots account is re-derived on-chain from the proof's timestamp — a spoofed account with a matching layout cannot be substituted.
4. **Two client auth modes.** The app ships a shared free-tier read token so judges run it instantly; the keeper can mint its own via the full on-chain activation flow (`apps/solana/keeper/src/txline.ts`).
5. **Mobile wallet duality.** A keychain-stored burner makes the demo deterministic; Mobile Wallet Adapter connects Phantom/Solflare for real self-custody. Both feed the same `signAndSend` interface.

## Known trade-offs

- kUSD is a demo SPL mint (faucet), standing in for USDC — swap the mint address to go real.
- Mobile polls scores every 6s on the match screen (RN streaming quirks); the keeper + dashboard consume the actual SSE stream.
- 1X2 goals only for now — `validate_stat_v2`'s multi-stat strategies (corners, cards, per-half) map cleanly onto prop-bet pools; the IDL types are already in the client.
